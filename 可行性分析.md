# 自动开发迭代方案 — 可行性分析

## 一、原方案核心流程回顾

```
修改代码 → Git Push → Jenkins 构建 → 上传 Skill/刷新 MCP → 触发测试 → 测试服务推送报告至 Git → Coding Webhook 回调 Agent → Agent 分析报告 → 循环迭代
```

原方案假设 **Agent 能够直接接收外部 HTTP Webhook 回调**（步骤 6），这是整个闭环的关键节点，也是最大的可行性疑问点。

---

## 二、核心问题：Agent 如何接收 Git Hook 通知

### 2.1 原方案实现方式

原方案在 `agent/webhook_server.py` 中启动了一个本地 HTTP 服务器（监听 `0.0.0.0:8888`），并计划将 `http://agent-host:8888/callback` 作为 Coding Service Hook 的回调 URL。

### 2.2 可行性评估

| 方案 | 工具 | 能否直接接收 Webhook | 原因 | 可行替代方案 |
|------|------|---------------------|------|-------------|
| **Cursor Agent** | Cursor IDE | **不能直接接收** | Cursor Agent 模式没有对外暴露 HTTP 服务端口的机制，且 IDE 运行在本地/远程开发机，无公网入口 | 使用 ngrok/Cloudflare Tunnel 暴露本地端口；或改用轮询方式 |
| **Claude Code SDK** | `claude-agent-sdk` | **不能直接接收** | SDK 提供的是程序化调用 Claude 的能力（`query()` / `ClaudeSDKClient`），本身不运行常驻 HTTP 服务 | 在 Python 脚本中自行启动 `http.server` 或 Flask 服务，配合内网穿透 |
| **通用方案** | Python 脚本 | **可以实现，但需额外配置** | 可以在 Agent 脚本中启动 HTTP 服务器，但开发机通常无公网 IP，需要内网穿透 | **ngrok、Cloudflare Tunnel、localtunnel** |

### 2.3 结论

> **Agent 本身无法直接接收外部 Webhook，必须借助内网穿透工具或改为轮询模式。**

## 三、待确认能否执行的实现点

### 3.1 已确认可行（有官方支持）

| 实现点 | 可行性 | 说明 |
|--------|--------|------|
| 使用 `claude-agent-sdk` 程序化修改代码 | **可行** | Anthropic 官方提供 Python SDK，支持 `query()` 和 `ClaudeSDKClient`，可自动执行文件读写和 Bash 命令 |
| Jenkins REST API 触发构建/查询状态 | **可行** | Jenkins 官方 Remote Access API 完善，支持 `POST /job/{name}/build`、轮询队列状态、获取控制台日志 |
| Coding Git 仓库 Webhook 配置 | **可行** | 支持 Push/MR 事件，可配置回调 URL、Token 签名验证 |
| Python 测试服务异步执行 pytest | **可行** | Flask/FastAPI 启动 HTTP 服务，多线程执行测试，标准做法 |

### 3.2 待确认/存在风险的实现点

| 实现点 | 风险等级 | 待确认事项 |
|--------|---------|-----------|
| **Agent 接收 Coding Webhook** | **高** | 开发机是否有公网 IP？是否允许使用 ngrok 等穿透工具？公司网络安全策略是否限制？ |
| **kt-agent 平台接口** | **中** | `POST /api/skills/upload` 和 `POST /api/mcp/refresh` 是否为真实存在的接口？需要 kt-agent 平台提供 API 文档 |
| **Jenkins 通过 commit hash 查找构建** | **中** | Jenkins API 默认不直接支持按 commit hash 查询构建，需要遍历构建列表比对 `lastBuiltRevision.SHA1`，性能较差；或改用 Jenkins 构建参数传递 commit hash |
| **测试服务推送报告至 Coding** | **中** | 测试服务需要具备 Git 仓库写权限，且频繁提交报告文件会污染仓库历史；建议确认是否接受此方案，或改为上传到对象存储（COS/OSS） |
| **Claude Code SDK 自动接受修改** | **中** | `permission_mode="acceptEdits"` 会自动应用所有文件修改，存在误改风险；建议确认是否允许全自动无人工审核 |
| **Jenkins 自动部署到生产/测试环境** | **高** | 原方案中 Jenkins Pipeline 包含 `scp` + `systemctl restart` 直接重启服务，需要确认是否允许 Agent 触发生产环境部署 |

### 3.3 确认清单（需与相关团队核实）

- [ ] **网络策略**：是否允许开发机使用 ngrok/Cloudflare Tunnel 暴露端口接收 Webhook？
- [ ] **kt-agent 平台**：确认 Skill 上传和 MCP 刷新接口的真实路径、认证方式、请求格式
- [ ] **Jenkins 权限**：Agent 使用的 Jenkins API Token 是否有权限触发指定 Job？
- [ ] **Git 权限**：测试服务使用的 Git 凭据是否有权限推送报告到 Coding 仓库？
- [ ] **部署权限**：Jenkins 是否有权限通过 SSH 部署到 MCP 服务器？是否允许自动重启服务？
- [ ] **安全审核**：是否接受 Claude Code 全自动修改代码并提交，无需人工 Review？

## 五、推荐替代方案：轮询模式（规避 Webhook 接收问题）

如果内网穿透不可行，建议将步骤 5-7 改为**轮询模式**：

```
步骤 5: 触发异步测试 → 获得 task_id
步骤 6: 轮询测试服务 /test/status/{task_id} 获取状态
步骤 7: 状态为 completed 后，主动下载报告并分析
```

### 5.1 轮询模式 vs Webhook 模式对比

| 维度 | Webhook 模式（原方案） | 轮询模式（推荐替代） |
|------|----------------------|---------------------|
| 网络要求 | 需要公网入口或内网穿透 | 仅需出网权限（通常开发机都有） |
| 实时性 | 实时推送 | 依赖轮询间隔（建议 5-10 秒） |
| 复杂度 | 高（需配置穿透、防火墙） | 低（纯 HTTP 客户端轮询） |
| 可靠性 | 依赖穿透服务稳定性 | 更稳定，不依赖第三方穿透 |
| 安全性 | 暴露端口存在风险 | 无端口暴露，更安全 |

### 5.2 修改后的流程

```
1. 修改代码 → 2. Git Push → 3. Jenkins 构建 → 4. 上传 Skill/刷新 MCP → 5. 触发测试（获得 task_id）
                                                                                       ↓
6. 轮询 /test/status/{task_id} ←────────────────────────────────────────────────────────┘
   状态 = completed?
   ├─ 否 → 等待 5 秒后继续轮询
   └─ 是 → 下载报告 → 7. 分析报告 → 决策是否继续迭代
```

## 六、总结与建议

| 问题 | 结论 | 建议 |
|------|------|------|
| Agent 能否直接接收 Webhook？ | **不能**，Cursor/Claude Code SDK 均无内置 HTTP 服务端 | 使用 ngrok 等穿透工具，或改用轮询模式 |
| 原方案整体是否可行？ | **基本可行，但需调整** | 将 Webhook 回调改为轮询，降低网络和部署复杂度 |
| 最大风险点 | kt-agent 平台接口未确认、自动部署生产环境权限 | 优先与平台团队确认 API，明确部署权限边界 |
| 最小可行实现（MVP） | 跳过自动部署，仅做代码修改 + Jenkins 编译验证 + 测试报告分析 | 先验证核心闭环，再逐步增加平台对接和自动部署 |
